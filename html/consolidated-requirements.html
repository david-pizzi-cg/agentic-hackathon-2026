Synthetic Data Strategy

  • Use Synthea or hand-crafted scenarios to generate realistic patient records
  • Minimum dataset: 20 patients with varied histories to demonstrate all scenarios
  • Must include cases that trigger: different urgency levels, referrals, appointment types
  • All data clearly marked as synthetic — no ambiguity

5. Non-Functional Requirements

Privacy & Compliance

  • No real PII/PHI at any stage — synthetic data only (hackathon)
  • Design patterns assume NHS/HIPAA-level sensitivity
  • All data encrypted at rest (Dataverse) and in transit
  • Role-based access: clinicians see patient data; patients see only their own; admin sees operational data

Responsible AI

  • Agents handle administrative tasks only — never make clinical decisions or diagnoses
  • Triage classifies urgency but does not diagnose — always directs to clinician or 999
  • Documentation structures clinician's notes — does not add clinical content
  • Clinician approval required before any record is saved or referral sent
  • Emergency triage always directs to 999 — no exceptions

Resilience

  • High Triage — must have defined fallback to phone/reception
  • Medium Documentation, Referral, Scheduling — manual fallback acceptable
  • Graceful degradation over hard failure
  • Patient-facing agents must clearly communicate when they cannot help
  • Clinician-facing agents must never block a consultation

Audit & Governance

  • Every agent action logged: agent ID, action type, entity affected, timestamp
  • Audit trail persists independently of agent availability
  • Clinical Lead and Admin can review agent activity via dashboard
  • Triage volumes, booking rates, referral completeness tracked for quality improvement
