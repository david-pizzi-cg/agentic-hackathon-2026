3. Key Scenarios (End-to-End Workflows)

Scenario 1: Consultation with Real-Time Decision Support

Trigger: Clinician starts a Teams consultation with a patient
  1. Documentation Agent joins the call, begins transcribing
  2. CDS Agent retrieves patient record from Dataverse — surfaces allergies, current medications, recent encounters in the Teams side panel
  3. Clinician discusses symptoms; CDS Agent matches symptoms to NICE guidelines, displays relevant pathways
  4. Clinician considers prescribing; CDS Agent flags a potential interaction with existing medication (severity: high, source: BNF)
  5. Clinician adjusts prescription based on alert
  6. Consultation ends; Documentation Agent generates structured summary
  7. Clinician reviews, edits, and approves notes
  8. CDS Agent runs post-consultation review — no gaps found
  9. Approved notes trigger Referral Agent to draft a referral letter
  10. Clinician reviews and sends referral; patient notified via chatbot
Personas: Clinician, Patient Agents: Documentation, CDS, Referral KPIs: Time saved, drug alerts, guideline adherence

Scenario 2: Patient Self-Service Triage and Booking

Trigger: Patient accesses the practice chatbot feeling unwell
  1. Triage Agent greets patient, asks structured symptom questions (empathetic, plain language)
  2. Patient describes chest pain and shortness of breath
  3. Triage Agent classifies as emergency — advises calling 999, provides clear instructions
  4. (Alternative: non-urgent) Triage Agent classifies as routine — asks about preferences
  5. Scheduling Agent shows available GP slots; patient selects and confirms
  6. Confirmation sent; encounter pre-populated with triage summary for the clinician
  7. Triage data logged for demand planning
Personas: Patient, Admin Agents: Triage, Scheduling KPIs: Triage accuracy, time to appointment, phone reduction

Scenario 3: Missed Referral Catch

Trigger: Post-consultation async review
  1. Documentation Agent has finalised consultation notes
  2. CDS Agent reviews notes against NICE guidelines
  3. CDS Agent identifies symptoms + diagnosis match a referral pathway (e.g. suspected cancer two-week-wait) but no referral was initiated
  4. Alert sent to clinician via Outlook: "Based on [guideline], a referral to [specialty] may be indicated. Review recommended."
  5. Clinician reviews, agrees, initiates referral
  6. Referral Agent drafts letter; clinician approves; patient notified
  7. Event logged in audit trail
Personas: Clinician, Patient, Clinical Lead Agents: Documentation, CDS, Referral KPIs: Missed referrals caught, time to referral, compliance

Scenario 4: Referral Status Check (Patient-Initiated)

Trigger: Patient messages the chatbot asking about their referral
  1. Patient asks: "What's happening with my referral?"
  2. Referral Agent queries Dataverse for referral status
  3. Responds in plain language: "Your referral to cardiology was sent on 10 March. The hospital has acknowledged it and you should receive an appointment letter within 2 weeks."
  4. Patient asks follow-up; agent answers or escalates to admin if outside scope
  5. Interaction logged
Personas: Patient Agents: Referral (via Triage chatbot) KPIs: Queries resolved without phone call, satisfaction

Agent Communication

Agents communicate through a central orchestration layer (Copilot Studio), not point-to-point:

Documentation Agent → [structured notes] → Orchestrator → Clinical Decision Support Agent
CDS Agent → [recommendations] → Orchestrator → Clinician (Teams)
CDS Agent → [referral recommendation] → Orchestrator → Referral Agent
Triage Agent → [urgency + routing] → Orchestrator → Scheduling Agent
Scheduling Agent → [confirmation] → Orchestrator → Patient (chatbot)

Synthetic Data Strategy

  • Use Synthea or hand-crafted scenarios to generate realistic patient records
  • Minimum dataset: 20 patients with varied histories to demonstrate all scenarios
  • Must include cases that trigger: drug interactions, missed referrals, emergency triage, routine bookings
  • All data clearly marked as synthetic — no ambiguity

6. Non-Functional Requirements

Privacy & Compliance

  • No real PII/PHI at any stage — synthetic data only (hackathon)
  • Design patterns assume NHS/HIPAA-level sensitivity
  • All data encrypted at rest (Dataverse) and in transit
  • Role-based access: clinicians see patient data; patients see only their own; admin sees operational data

Responsible AI

  • Agents present evidence with cited sources — never state diagnoses
  • Confidence indicators on all suggestions (high/medium/low)
  • Clinician override always available and logged
  • No autonomous clinical decisions — every suggestion requires human approval
  • Bias monitoring: track recommendation patterns across demographics

Resilience

  • High Triage, CDS — must have defined fallback to human
  • Medium Documentation, Referral, Scheduling — manual fallback acceptable
  • Graceful degradation over hard failure
  • Patient-facing agents must clearly communicate when they cannot help
  • Clinician-facing agents must never block a consultation

Audit & Governance

  • Every agent action logged: agent ID, action type, entity affected, clinician decision, timestamp
  • Audit trail persists independently of agent availability
  • Clinical Lead can review all agent activity via dashboard
  • Override rates and guideline adherence tracked for quality improvement
